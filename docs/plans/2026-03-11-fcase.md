# fcase Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Implement `fcase` as the PRD-defined continuity and handoff layer for fsuite investigations, with current-state reads, typed targets/evidence/hypotheses, handoff output, and a portable export envelope.

**Architecture:** Build `fcase` as a new Bash CLI backed by a dedicated SQLite database at `~/.fsuite/fcase.db`. Keep the primary read model in normalized tables (`cases`, `case_sessions`, `targets`, `evidence`, `hypotheses`) and use an append-only `events` table for history and audit only. The MVP ships the thin ledger first and leaves explicit room for future `fread`/`fmap` imports without letting imports dominate the base implementation.

**Tech Stack:** Bash, SQLite via `sqlite3`, shell JSON formatting, fsuite shell test harnesses, Python 3 for JSON assertions where needed

**Step 3: Write minimal implementation**

Implement:

- `target add`
- `evidence`
- `hypothesis add`
- `hypothesis set`
- `reject`

Rules:

- target states limited to `candidate`, `active`, `validated`, `ruled_out`
- hypothesis statuses limited to `open`, `active`, `validated`, `rejected`
- `reject` requires exactly one selector:
  - `--target-id <id>`
  - `--hypothesis-id <id>`
- `reject` only maps to typed state transitions
- `evidence` requires exactly one body source:
  - `--body`
  - `--body-file`
- `--lines` must be `start:end`, 1-based inclusive, positive integers, `start <= end`
- `--match-line` must fall within `--lines` when both are present
- append corresponding `events` rows for every state transition
